# CI/CD Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers-extended-cc:subagent-driven-development (recommended) or superpowers-extended-cc:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire up GitHub Actions CI (build + test on push/PR) and release (publish to NuGet.org on `v*` tag push, via OIDC trusted publishing), and prove it end-to-end with a `v0.1.0` release.

**Architecture:** Two workflow files under `.github/workflows/`. CI workflow runs on push to `main` and pull requests. Release workflow runs on `v*` tag push, re-runs build + tests, packs, then exchanges a GitHub OIDC token for a short-lived NuGet API key via `NuGet/login@v1` and pushes the `.nupkg` (which carries the `.snupkg` symbol package alongside). NuGet lockfiles are enabled repo-wide so the release restore can use `--locked-mode` and CI cache keys are stable.

**Tech Stack:** GitHub Actions, `actions/checkout@v4`, `actions/setup-dotnet@v4`, `NuGet/login@v1`, .NET 10 SDK, MinVer (already in place), NuGet.org trusted publishing.

**Steps:**

- [ ] **Step 1: Create the workflows directory**

```bash
mkdir -p .github/workflows
```

- [ ] **Step 2: Create `.github/workflows/ci.yml`**

```yaml
name: CI

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: ${{ github.event_name == 'pull_request' }}

permissions:
  contents: read

jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-dotnet@v4
        with:
          global-json-file: global.json
          cache: true
          cache-dependency-path: '**/packages.lock.json'

      - run: dotnet restore
      - run: dotnet build -c Release --no-restore
      - run: dotnet test -c Release --no-build --logger "GitHubActions;summary.includePassedTests=false"
```

Notes for the implementer:
- The file MUST be exactly this — no extra whitespace tweaks, no editor-injected trailing spaces beyond what `.editorconfig` allows. GitHub's YAML parser is unforgiving.
- The `GitHubActions` test logger is built into `Microsoft.NET.Test.Sdk` 17.x (which is what the foundation references). No additional package needed.
- CI does **not** use `--locked-mode` on restore — drift surfaces as a build warning during PR review, not a hard failure. The release workflow (Task 3) is the strict-mode path.
- `cache-dependency-path` glob will match the five lockfiles created in Task 1.

### Task 3: Release workflow (publish on tag push)

**Goal:** Create `.github/workflows/release.yml` that fires on `v*` tag push, re-runs build + tests under `--locked-mode`, packs the main lib, exchanges OIDC for a NuGet API key, and pushes the `.nupkg` (which uploads the sibling `.snupkg` automatically).

**Files:**
- Create: `.github/workflows/release.yml`

**Acceptance Criteria:**
- [ ] `.github/workflows/release.yml` exists with the exact content below.
- [ ] YAML is syntactically valid.
- [ ] Trigger is `push.tags: ['v*']` only — no other triggers.
- [ ] `permissions:` block includes `id-token: write` (required for OIDC exchange) and `contents: read`.
- [ ] Job runs in the `nuget-release` GitHub Deployment Environment.
- [ ] `dotnet restore` uses `--locked-mode`.
- [ ] `dotnet pack` uses `--no-build` to reuse the verified build artifacts.
- [ ] `dotnet nuget push` uses the `*.nupkg` glob and passes `--skip-duplicate`.

**Verify:** `python3 -c "import yaml,sys; yaml.safe_load(open('.github/workflows/release.yml')); print('ok')"` → prints `ok`.

**Steps:**

- [ ] **Step 1: Create `.github/workflows/release.yml`**

```yaml
name: Release

on:
  push:
    tags: ['v*']

permissions:
  contents: read
  id-token: write

jobs:
  publish:
    runs-on: ubuntu-latest
    environment: nuget-release
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-dotnet@v4
        with:
          global-json-file: global.json
          cache: true
          cache-dependency-path: '**/packages.lock.json'

      - run: dotnet restore --locked-mode
      - run: dotnet build -c Release --no-restore
      - run: dotnet test -c Release --no-build --logger "GitHubActions;summary.includePassedTests=false"

      - run: dotnet pack -c Release --no-build -o ./artifacts src/JoakimAnder.Toolbox/JoakimAnder.Toolbox.csproj

      - uses: NuGet/login@v1
        id: nuget-login
        with:
          api-key-source-url: https://api.nuget.org/v3/index.json

      - run: >
          dotnet nuget push ./artifacts/*.nupkg
          --api-key "${{ steps.nuget-login.outputs.NUGET_API_KEY }}"
          --source https://api.nuget.org/v3/index.json
          --skip-duplicate
```

Notes for the implementer:
- `id-token: write` is mandatory for OIDC. Without it, `NuGet/login@v1` cannot obtain a GitHub OIDC token to exchange for a NuGet API key.
- `environment: nuget-release` makes the run show up as a real "deployment" in the repo timeline and is also the field the trusted-publisher policy on nuget.org will match against (Task 5).
- The `*.nupkg` glob in the push command matches only the main package. `dotnet nuget push` automatically uploads any sibling `.snupkg` (the `IncludeSymbols=true` + `SymbolPackageFormat=snupkg` from the foundation's `src/Directory.Build.props` produces one), so symbols ship without an extra command.
- `--skip-duplicate` makes the push idempotent: if the workflow is re-run for the same tag (e.g. nuget.org transient error), nuget.org returns 409 "already exists" and the workflow treats it as success rather than failing the run.
- Tests are deliberately re-run before publish, even though CI already ran them on the commit being tagged. This guards the case where a tag is pushed at an older commit than CI's latest green run.
